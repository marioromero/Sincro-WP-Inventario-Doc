01 · Visión General

Contexto del Producto

Sistema POS + Inventario multi-sucursal diseñado para el mercado chileno, operando como capa intermedia entre WordPress + WooCommerce y los puntos de venta físicos.

¿Por qué este producto?

Los comercios chilenos que operan con WooCommerce enfrentan una desconexión crítica entre su tienda online y sus sucursales físicas: el stock no se sincroniza en tiempo real, las ventas presenciales no se reflejan en el e-commerce, y no existe una fuente de verdad única para el inventario. Este sistema resuelve ese vacío actuando como el sistema de registro central que orquesta tanto los canales digitales como los físicos.

Mercado objetivo

  • Pymes y mid-market chileno con 2 a 20 sucursales físicas.
  • Comercios que ya usan WooCommerce y necesitan un POS moderno conectado.
  • Dueños de negocio que requieren visibilidad consolidada de inventario y ventas.

Diferenciadores clave

  • Local-First: El sistema funciona incluso si la conexión a internet cae, gracias al motor de colas con reintentos.
  • Multi-sucursal nativo: Cada sucursal opera con su propio perfil de stock, precios y configuración.
  • Zero CORS: Al usar Inertia.js sobre Laravel, el frontend y backend comparten el mismo origen, eliminando problemas de CORS en hosting compartido.
  • Sin demonios: Diseñado para funcionar en cPanel sin procesos persistentes; las tareas diferidas se ejecutan mediante cron.
Filosofía de diseño: Preferimos un monolito bien modularizado sobre una arquitectura de microservicios. La razón es operativa: el cliente chileno típico contrata hosting compartido donde no puede administrar contenedores ni orquestadores. Un monolito Laravel con boundaries de dominio claros es desplegable, depurable y mantenible por un solo desarrollador.
01 · Visión General

Modelo Standalone (Local-First)

El sistema opera como una aplicación independiente, no como un plugin de WordPress. Esta decisión arquitectónica es fundamental.

¿Qué significa "Standalone"?

El sistema corre en su propio subdominio (ej. sistema.mi-tienda.cl) con su propia base de datos Laravel. WordPress y WooCommerce quedan en otro subdominio o ruta (ej. tienda.mi-tienda.cl o /tienda). La comunicación entre ambos sistemas ocurre exclusivamente por API.

Ventajas del modelo Standalone

  • Aislamiento tecnológico: Laravel no comparte el proceso de PHP con WordPress. Una actualización de WooCommerce no rompe el POS.
  • Stack moderno: Podemos usar Vue 3, Tailwind y Vite sin depender de la cadena de build de WordPress.
  • Escalabilidad independiente: Cada sistema puede escalar sus recursos según su demanda específica.
    • Seguridad: El panel administrativo no expone rutas de WordPress. Se reduce la superficie de ataque.

Arquitectura Local-First

A diferencia de las SaaS que requieren conexión permanente, este sistema prioriza la operación local:

  1. Operación local inmediata: Cada transacción POS se registra primero en la base de datos local de Laravel.
  2. Sincronización asíncrona: Las actualizaciones hacia WooCommerce se encolan y procesan sin bloquear al usuario.
  3. Reintentos automáticos: Si WooCommerce no responde, la cola reintenta con backoff exponencial.
  4. Consistencia eventual: El sistema garantiza que el stock de WooCommerce convergerá al valor real, pero acepta desfases temporales controlados.
Importante: El modelo Local-First implica que WooCommerce no es la fuente de verdad del stock. La fuente de verdad es el Stock Ledger del POS. WooCommerce es un reflejo (copia) cuyo rezago máximo esperado es de 1-2 minutos en condiciones normales.

02 · Arquitectura

Stack Tecnológico

Justificación completa de cada decisión tecnológica, con énfasis en las restricciones del entorno cPanel de hosting compartido chileno.

Backend: Laravel 11

Laravel es la opción natural para un monolito PHP moderno. Proporciona ORM (Eloquent), sistema de colas con múltiples drivers, autenticación, autorización por políticas, y un ecosistema maduro de paquetes. La versión 11 trae mejoras de rendimiento significativas y una estructura más limpia.

Frontend: Vue 3 + Inertia.js

Justificación clave: En un entorno cPanel, donde no podemos configurar un proxy reverso ni manejar CORS fácilmente, Inertia.js elimina la necesidad de una API REST separada. El frontend Vue 3 se renderiza desde las mismas rutas de Laravel, compartiendo sesión, CSRF y autenticación sin configuración adicional.

Alternativa descartada — API stateless (Breeze/Sanctum): Requeriría manejar tokens JWT, refresh tokens, CORS headers y una doble pipeline de despliegue (frontend separado). En cPanel, cada origen extra es una fuente potencial de bloqueos y bugs difíciles de depurar. Inertia.js reduce la superficie de integración a una sola capa.

Base de Datos: MySQL 8 / MariaDB 10.6+

Motor InnoDB con transacciones ACID. El esquema usa migraciones de Laravel. Se prioriza la integridad referencial sobre la flexibilidad documental (no usamos MongoDB ni bases文档). El driver de colas será database (tabla jobs), no Redis, porque cPanel no ofrece Redis como servicio estándar.

Build Tooling: Vite + Laravel Vite Plugin

Vite reemplaza a Laravel Mix (Webpack). Ofrece HMR ultrarrápido en desarrollo y builds optimizadas en producción. Los assets compilados se sirven desde public/build/.

Cron como Task Scheduler

En cPanel no podemos mantener un proceso supervisor (como Supervisor en Linux) para gestionar workers de cola. La solución es un cron maestro que ejecuta php artisan queue:work --stop-when-empty cada minuto. Este patrón es estándar en la comunidad Laravel para entornos compartidos.

02 · Arquitectura

Infraestructura cPanel

Restricciones operativas del entorno de hosting compartido y cómo la arquitectura se adapta a ellas.

Restricciones conocidas de cPanel

  • Sin demonios persistentes: No podemos ejecutar php artisan queue:work como un servicio supervisor. No hay systemd, supervisord ni screen/tmux disponible.
  • Cron con intervalo mínimo de 1 minuto: El scheduler de cPanel permite ejecuciones tan frecuentes como cada minuto. Esto define el límite inferior de latencia de nuestras colas.
  • Sin Redis/Memcached garantizado: La mayoría de planes compartidos no incluyen Redis. Algunos ofrecen Memcached pero con recursos limitados. Usamos la base de datos como driver de cola por defecto.
  • PHP-FPM con límites de procesos: El número de workers PHP está restringido. Laravel Octane (Roadrunner/Swoole) no es viable. Mantenemos el ciclo de vida request-response tradicional.
  • Sin control de servidor web: No podemos configurar Nginx personalizado ni reglas avanzadas de Apache. El archivo .htaccess es nuestra única herramienta de reescritura.
  • Almacenamiento compartido: El disco es típicamente NFS o similar. Las operaciones de I/O intensivas deben minimizarse.

Estrategia de adaptación

RestricciónSolución
Sin demoniosCron ejecuta php artisan queue:work --stop-when-empty cada minuto
Sin RedisDriver de cola database con tabla jobs indexada
PHP-FPM limitadoPeticiones Inertia ligeras; API calls externalizadas a jobs de cola
Sin control de servidorToda la lógica de ruteo vive en Laravel, no en el servidor web
I/O compartidaSubida de archivos a disco local con sync periódico a cloud storage

Configuración recomendada de hosting

  • PHP 8.2+ con extensiones: BCmath, Ctype, JSON, Mbstring, OpenSSL, PDO, Tokenizer, XML, GD, Zip
  • MySQL 8+ con al menos 1 GB de espacio para la base de datos
  • Node.js 20+ (para compilación de assets con Vite)
  • Composer 2.x para gestión de dependencias PHP
  • Subdominio dedicado (ej. sistema.tienda.cl) con SSL
Atención: No todos los planes de hosting compartido chileno soportan Node.js. Si el plan no lo incluye, los assets deben compilarse en entorno local y subirse ya build (carpeta public/build/). Es una limitación conocida que debe validarse antes de contratar.
02 · Arquitectura

Diagrama de Topología

Visión macro del despliegue: cómo se conectan el navegador del usuario, el monolito Laravel en cPanel y WooCommerce.

Flujo de datos explicado

  1. Navegación del usuario: El operador POS accede mediante Inertia.js. Las vistas se renderizan desde Laravel (SSR híbrido) sin llamadas API separadas.
  2. Operación transaccional: Cada venta POS impacta primero la base de datos local (Stock Ledger). No depende de WooCommerce para completarse.
  3. Encadenamiento asíncrono: Las operaciones que deben reflejarse en WooCommerce se despachan como Jobs a la cola database.
  4. Procesamiento por Cron: Cada minuto, el cron de cPanel ejecuta php artisan queue:work --stop-when-empty, procesando todos los jobs pendientes.
  5. Sincronización saliente: Los jobs ejecutan llamadas HTTP a la API REST de WooCommerce para actualizar stock, precios o crear productos.
  6. Recepción de cambios: WooCommerce notifica eventos (nuevo pedido web, cambio de stock) vía webhook POST hacia el controlador correspondiente.
  7. Consistencia eventual: El sistema garantiza que el estado de WooCommerce convergerá al estado real, con un rezago máximo esperado de ~1-2 minutos.
02 · Arquitectura

Contrato de Licenciamiento

Mecanismo de licenciamiento diseñado para el MVP: un stub local que siempre responde como activo, con una interfaz preparada para reemplazo futuro por una API real de validación.

Filosofía: habilitar, no restringir

En la fase MVP, el objetivo es maximizar la adopción y reducir la fricción de instalación. Implementar un sistema de licenciamiento remoto desde el día cero agregaría complejidad operativa (el sistema no funcionaría sin internet), dependencia de un servicio externo aún no construido, y una experiencia de onboarding negativa.

Por eso, el MVP incluye un Licensing Stub: un contrato (interface) que el sistema consulta para verificar si la licencia está activa, pero cuya implementación concreta es una clase local que siempre retorna true.

Diseño del contrato

// App\Contracts\LicenseValidator.php
<?php

namespace App\Contracts;

interface LicenseValidator
{
    public function isValid(string $licenseKey): bool;
    public function getLicenseInfo(string $licenseKey): array;
    public function getDaysRemaining(string $licenseKey): ?int;
}

Implementación concreta (MVP)

// App\Services\LocalLicenseStub.php
<?php

namespace App\Services;

use App\Contracts\LicenseValidator;

class LocalLicenseStub implements LicenseValidator
{
    public function isValid(string $licenseKey): bool
    {
        return true; // Stub: siempre activo en MVP
    }

    public function getLicenseInfo(string $licenseKey): array
    {
        return [
            'status'    => 'active',
            'plan'      => 'mvp-standalone',
            'source'    => 'stub',
        ];
    }

    public function getDaysRemaining(string $licenseKey): ?int
    {
        return 365; // Stub: simula 1 año
    }
}

Estrategia de reemplazo futuro

Cuando el producto escale a un modelo SaaS o se requiera validación centralizada:

  1. Crear App\Services\CloudLicenseService.php que implemente LicenseValidator, conectándose a una API remota de licencias.
  2. Cambiar el binding en AppServiceProvider:
// app/Providers/AppServiceProvider.php

// MVP:
\$this->app->bind(LicenseValidator::class, LocalLicenseStub::class);

// Versión SaaS futura (solo cambiar esta línea):
// \$this->app->bind(LicenseValidator::class, CloudLicenseService::class);

El resto del código no requiere modificaciones. Todos los módulos que necesitan verificar licencia dependen de la interfaz LicenseValidator, no de una implementación concreta.

Principio aplicado: Programación orientada a interfaces (Dependency Inversion). El dominio no debe conocer los detalles de infraestructura. El stub y la API futura son intercambiables sin modificar una sola línea de lógica de negocio.
03 · Sincronización WooCommerce

Reglas de Verdad

Definición precisa de qué sistema es la fuente autoritativa para cada fragmento de datos, y cómo se resuelven los conflictos cuando ambos sistemas modifican el mismo recurso.

Stock: bidireccional con jerarquía POS

El stock es bidireccional en el sentido de que ambos sistemas pueden modificarlo (WooCommerce descuenta stock cuando llega un pedido web, el POS descuenta cuando hay venta presencial), pero el POS siempre impone su valor sobre WooCommerce cuando hay discrepancia sobre la disponibilidad física local.

  • Venta POS: descuenta del Stock Ledger local y encola sync hacia WooCommerce.
  • Pedido web: WooCommerce descuenta primero; al importar el pedido, el POS verifica y ajusta si el stock web estaba incorrecto.
  • Ajuste de inventario: Siempre se hace en el POS y se propaga a WooCommerce. No se aceptan ajustes desde WooCommerce.

Catálogo: bidireccional con resolución de conflictos

El catálogo de productos puede modificarse tanto desde el POS como desde WooCommerce. El desarrollador debe elegir una estrategia de resolución de conflictos e implementarla de forma consistente. Se documentan dos estrategias viables:

Estrategia A: "Gana el timestamp más reciente"

Cada modificación registra un campo updated_at. Al sincronizar, se compara el timestamp de ambos sistemas y el valor más reciente prevalece. Es la estrategia más simple pero puede perder cambios si hay diferencias de reloj entre servidores (>5s de skew puede causar regresiones).

// Pseudo-código: Estrategia A
function resolveConflict(Product \$local, array \$wooData): array
{
    \$localTime = \$local->updated_at->timestamp;
    \$wooTime = strtotime(\$wooData['date_modified']);

    if (\$localTime >= \$wooTime) {
        return \$local->toArray();  // POS gana
    }
    return \$wooData;  // WooCommerce gana
}

Estrategia B: "Bloqueo de campos según origen"

Cada campo del producto tiene un origen propietario definido en una tabla de mapeo. El POS es dueño de sku, stock y price. WooCommerce es dueño de description, images y categories. La sincronización respeta esa propiedad: nunca sobreescribe un campo cuyo origen es el otro sistema.

// Config: field_ownership.php
return [
    'sku'         => 'pos',
    'name'        => 'pos',
    'price'       => 'pos',
    'stock'       => 'pos',
    'description' => 'woocommerce',
    'images'      => 'woocommerce',
    'categories'  => 'woocommerce',
    'tags'        => 'woocommerce',
];
Trade-off: Estrategia A es más fácil de implementar y entender, pero vulnerable a desincronización de relojes. Estrategia B es más robusta pero requiere configuración explícita por campo y agregar un mecanismo de "reclamo de propiedad" si un operador necesita forzar la sobreescritura.

Gráfica de consistencia

El sistema opera en el espectro de consistencia eventual. No hay bloqueo distribuido ni transacciones distribuidas. La ventana de inconsistencia máxima esperada es:

  • Condiciones normales: < 1 minuto
  • Pico de carga (ej. CyberDay): < 5 minutos
  • Caída de WooCommerce: indeterminada (backoff exponencial con reintentos cada 5 min × hasta 24 hrs)
03 · Sincronización WooCommerce

Webhooks y Sincronización

Detalle del motor de sincronización: eventos entrantes (webhooks), operaciones salientes (REST API asíncrona), y mecanismos de resiliencia.

Integración Entrante: Webhooks (WooCommerce → POS)

WooCommerce dispara webhooks POST hacia rutas dedicadas del POS cuando ocurren eventos. El POS expone endpoints públicos protegidos por un secreto compartido (HMAC-SHA256). Cada webhook se procesa dentro de un Job para no bloquear la respuesta HTTP a WooCommerce.

Evento WooCommerceEndpoint POSAcción
order.createdPOST /api/webhooks/order-createdImportar pedido como venta en estado en_transito. El pedido queda pendiente de asignación de sucursal (ruta de despacho).
order.updatedPOST /api/webhooks/order-updatedActualizar estado y notas internas del pedido en POS. No modifica líneas de detalle ni cantidades.
product.updatedPOST /api/webhooks/product-updatedRe-sincronizar campos cuya propiedad pertenece a WooCommerce según la estrategia de resolución elegida (descripción, imágenes, categorías).
product.createdPOST /api/webhooks/product-createdImportar producto como "no vinculado" al POS. Un administrador debe mapearlo manualmente a un producto existente o aprobar su creación.
stock.updatedPOST /api/webhooks/stock-updatedRegistrar en log de auditoría. No sobreescribe el Stock Ledger del POS. El POS forzará su valor en la siguiente sincronización.

Validación de webhooks

// Middleware: VerifyWebhookSignature
public function handle(Request \$request, Closure \$next)
{
    \$signature = \$request->header('X-WC-Webhook-Signature');
    \$payload = \$request->getContent();
    \$expected = base64_encode(hash_hmac('sha256', \$payload, config('services.woocommerce.webhook_secret'), true));

    if (!hash_equals(\$expected, \$signature)) {
        abort(401, 'Invalid webhook signature');
    }

    return \$next(\$request);
}

Integración Saliente: REST API asíncrona (POS → WooCommerce)

Los cambios originados en el POS nunca llaman a la API de WooCommerce de forma síncrona durante una request HTTP del usuario. Siempre se despachan Jobs a la cola database, garantizando que la operación POS no depende de la disponibilidad ni latencia de WooCommerce.

Evento POSJobEndpoint WooCommercePrioridad

Actualizar stock por venta POSSyncStockToWooCommercePUT /wp-json/wc/v3/products/{id}Alta
Transferencia entre sucursalesSyncTransferToWooCommercePUT /wp-json/wc/v3/products/{id}Normal
Crear/actualizar clienteSyncCustomerToWooCommercePOST /wp-json/wc/v3/customersBaja

Resiliencia del motor de sincronización

  • Reintentos con backoff: 5 reintentos con backoff exponencial (30s, 2min, 5min, 15min, 30min).
  • Dead Letter Queue: Jobs que fallan después de 5 intentos se mueven a tabla failed_jobs. Un comando artisan (php artisan sync:retry-all) permite re-encolarlos manualmente.
  • Idempotencia: Cada job incluye un UUID en el header X-Idempotency-Key. WooCommerce almacena esta clave por 24h para evitar duplicados si el job se ejecuta más de una vez.
  • Throttling: Respetamos los límites de rate de la API (usualmente 25 req/s). Se usa el middleware throttle de Laravel HTTP Client con un semáforo por CanalWoo.
  • Cola por prioridad: Los jobs de stock usan la cola high; los de clientes, low. El cron procesa primero high antes de tocar low.
03 · Sincronización WooCommerce

Reconciliación de Stock

Job programado que cruza el stock total del POS contra el stock registrado en WooCommerce para detectar discrepancias y forzar la sincronización.

¿Por qué es necesaria?

Incluso con un motor de sincronización robusto, pueden ocurrir discrepancias: un webhook no entregado, un job que falló y se perdió en la Dead Letter Queue, una modificación manual directa en WooCommerce, o un error de red transitorio que dejó una operación a medio camino. La reconciliación es la red de seguridad que detecta y corrige estos desajustes.

Diseño del Job de Reconciliación

Se implementa como un comando Artisan (php artisan sync:reconcile-stock) que se programa en el cron de cPanel para ejecutarse en ventanas de bajo tráfico (ej. 3:00 AM).

// app/Console/Commands/ReconcileStock.php
public function handle()
{
    // 1. Obtener todos los productos activos con mapeo WooCommerce
    \$products = Product::whereHas('wooMapping')->get();

    foreach (\$products as \$product) {
        // 2. Obtener stock real desde el Ledger del POS
        \$posStock = \$product->getStockTotal();  // suma de ledger (todas las sucursales)

        // 3. Obtener stock desde API de WooCommerce
        \$wooStock = Http::withToken(\$product->canalWoo->api_key)
            ->get(\$product->canalWoo->api_url . '/products/' . \$product->wooMapping->woo_id)
            ->json('stock_quantity');

        // 4. Si discrepa, encolar job de corrección
        if (\$posStock !== \$wooStock) {
            Log::warning('Stock discrepancy detected', [
                'sku' => \$product->sku,
                'pos' => \$posStock,
                'woo' => \$wooStock,
            ]);
            SyncStockToWooCommerce::dispatch(\$product)
                ->onQueue('high');
        }
    }
}

Configuración del Cron

En cPanel, se agrega una entrada al cron job que ejecuta el comando una vez al día:

# Reconciliación diaria de stock — ejecutar a las 3:00 AM
0 3 * * * /usr/local/bin/php /home/user/public_html/artisan sync:reconcile-stock >> /dev/null 2>&1

Consideraciones de rendimiento

  • Para catálogos grandes (>10,000 productos), la llamada individual a la API de WooCommerce por producto es lenta. Alternativa: usar /wp-json/wc/v3/products?page=1&per_page=100 y construir un mapa en memoria.
  • El job respeta el throttle de 25 req/s de WooCommerce. Si hay muchos productos con discrepancias, se encolan gradualmente.
  • Se recomienda monitorear el log de discrepancias: un volumen alto sostenido indica un problema sistémico (webhook caído, error de configuración).
Nota: La reconciliación no reemplaza la sincronización en tiempo real. Es un complemento de verificación periódica. La sincronización vía webhooks + jobs sigue siendo el canal principal.
04 · Inventario y Catálogo

Stock Ledger

El Stock Ledger es el corazón del inventario: un registro inmutable y auditable de cada movimiento que afecta el stock de un producto en una sucursal. Prohibido usar un campo de texto plano para el stock actual.

Principios del Ledger

  • Inmutabilidad: Una vez registrado, un movimiento no se elimina ni modifica. Solo se compensa con un movimiento inverso (ej. una venta anulada genera un movimiento de reversión).
  • Auditabilidad: Cada movimiento captura: quién (user_id), qué (product_id, branch_id), cuándo (created_at), por qué (type enum + reason text), cuánto (quantity) y saldo resultante (balance_after).
  • Trazabilidad: Cada movimiento referencia el documento origen mediante reference_type y reference_id (polimórfico).

Tipos de Movimiento

El enum type clasifica cada operación. Todos los tipos son transaccionales y se registran dentro de una transacción de base de datos junto con el documento que los origina:

Esquema conceptual

Schema::create('stock_ledger', function (Blueprint \$table) {
    \$table->id();
    \$table->foreignId('product_id')->constrained();
    \$table->foreignId('branch_id')->constrained('sucursales');
    \$table->string('type');  // ver tabla de tipos arriba
    \$table->integer('quantity');  // positivo = entrada, negativo = salida
    \$table->integer('balance_after');  // saldo posterior, calculado al insertar
    \$table->foreignId('user_id')->constrained();
    \$table->nullableMorphs('reference');  // referencia polimórfica al documento origen
    \$table->text('reason')->nullable();
    \$table->timestamps();

    \$table->index(['product_id', 'branch_id', 'created_at']);
});

Cálculo del stock disponible

No se almacena un campo "stock" calculado. El stock se obtiene sumando la tabla ledger:

public function stockEnSucursal(Branch \$branch): int
{
    return (int) \$this->stockLedger()
        ->where('branch_id', \$branch->id)
        ->sum('quantity');
}

Para el stock "disponible real" se excluyen las reservas en tránsito:

public function stockDisponible(Branch \$branch): int
{
    return \$this->stockEnSucursal(\$branch)
        - (int) \$this->stockLedger()
            ->where('branch_id', \$branch->id)
            ->where('type', 'reserva_en_transito')
            ->sum('quantity');
}
Nota de rendimiento: Para inventarios de <100k movimientos por sucursal, la suma directa con índice compuesto es suficientemente rápida (<50ms). Para volúmenes mayores, implementar tabla de resumen (branch_product_stock) actualizada por eventos de dominio.
04 · Inventario y Catálogo

Sucursales y Canales Web

El modelo multi-sucursal se complementa con el concepto de "canal web": cada tienda WooCommerce conectada es un canal independiente con sus propias credenciales y configuración.

Sucursales (Branches)

Las sucursales se modelan como una entidad separada (branches), no como un simple campo de texto. Cada venta o movimiento pertenece siempre a una sucursal. La entidad almacena:

  • Datos de contacto (dirección, teléfono, email, comuna, región)
  • Horario de atención (JSON con rangos semanales y excepciones festivas)
  • Configuración POS (impresora por defecto, cajón de dinero, logo personalizado, formato de ticket)
  • Usuarios asignados (relación many-to-many con roles específicos por sucursal)
  • Precios propios (sobreescriben el precio base del producto vía tabla pivot branch_product)
  • Stock segregado (cada movimiento en el ledger pertenece a una sucursal)
  • Cobertura de despacho (relación con tabla de comunas para enrutamiento geográfico)

Canales Web (CanalWoo)

Un cliente puede tener múltiples tiendas WooCommerce (ej. una tienda retail, una tienda B2B, una tienda para Chile y otra para Perú). Cada una se modela como un CanalWoo independiente:

Schema::create('canal_woos', function (Blueprint \$table) {
    \$table->id();
    \$table->foreignId('branch_id')->constrained();  // sucursal default para pedidos
    \$table->string('nombre');  // ej. "Tienda Retail", "Tienda B2B"
    \$table->string('url');     // URL base de la tienda WooCommerce
    \$table->string('api_key'); // Consumer Key (cifrado en reposo)
    \$table->string('api_secret'); // Consumer Secret (cifrado en reposo)
    \$table->string('webhook_secret'); // Secreto para validar webhooks entrantes
    \$table->boolean('activo')->default(true);
    \$table->json('config')->nullable();  // mapeo de estados, cola de prioridad, etc.
    \$table->timestamps();
});

Cada producto puede tener un mapeo distinto a cada canal WooCommerce, ya que el ID del producto en WooCommerce difiere por tienda. Esto se modela con una tabla pivot:

Schema::create('product_woo_mappings', function (Blueprint \$table) {
    \$table->id();
    \$table->foreignId('product_id')->constrained();
    \$table->foreignId('canal_woo_id')->constrained();
    \$table->unsignedBigInteger('woo_id');  // ID del producto en WooCommerce
    \$table->string('woo_sku')->nullable(); // SKU en WooCommerce (puede diferir)
    \$table->json('meta_data')->nullable();  // atributos extra de WooCommerce
    \$table->timestamps();

    \$table->unique(['product_id', 'canal_woo_id']);
    \$table->unique(['canal_woo_id', 'woo_id']);
});

Multi-sucursal + Multi-canal: reglas de negocio

  • Un CanalWoo pertenece a una sucursal por defecto. Los pedidos web de ese canal se asignan inicialmente a esa sucursal (luego puede reasignarse por ruta de despacho).
  • Cada CanalWoo tiene sus propias credenciales API y su propio webhook secret. La sincronización itera por canal.
  • Un producto puede estar publicado en múltiples canales WooCommerce, cada uno con su propio ID remoto y posiblemente SKU diferente.
  • La reconciliación de stock se ejecuta por canal: el stock total del POS se distribuye según la configuración de cada canal (ej. Canal A recibe el 100% del stock de sucursal 1; Canal B recibe la suma de sucursales 1 y 2).

Transferencias entre sucursales

El flujo de transferencia sigue un proceso de dos pasos (origen y destino):

  1. Salida (origen): Se registra transfer_out en la sucursal A. El producto queda "en tránsito".
  2. Entrada (destino): Cuando la sucursal B confirma recepción, se registra transfer_in. Firma digital del receptor opcional.

05 · Ventas y Pedidos Web

Rutas de Despacho

Los pedidos web importados desde WooCommerce requieren asignación a una sucursal para su preparación y despacho. El MVP soporta dos modos de asignación, y un tercero queda en backlog.

Modo A: Sucursal fija / Bodega central

Configuración: Todo pedido web se asigna automáticamente a una sucursal específica (la "bodega central" o "sucursal de despacho") configurada en el CanalWoo.

Cuándo usarlo: Clientes con una sola sucursal operativa que centraliza todos los despachos, o clientes que tercerizan la logística y solo necesitan consolidar pedidos en un punto.

Implementación: El campo canal_woos.branch_id se usa como sucursal de despacho default. No hay intervención manual.

Modo B: Selección manual por supervisor

Configuración: Los pedidos web llegan en estado pending_assignment y aparecen en un "buzón de asignación" del panel POS. Un supervisor selecciona manualmente la sucursal de preparación y despacho.

Cuándo usarlo: Clientes con múltiples sucursales donde la decisión de despacho depende de factores no modelados (disponibilidad de personal, horarios, tipo de producto).

Implementación: El supervisor ve una lista de pedidos no asignados con datos del cliente y productos. Selecciona una sucursal y el pedido pasa a estado assigned.

Modo C: Enrutamiento geográfico (backlog Fase 2)

Concepto: El sistema asigna automáticamente el pedido a la sucursal más cercana al domicilio del cliente según la comuna de envío, usando una tabla de cobertura por sucursal.

Por qué no está en el MVP: Requiere mantener una base de datos de comunas y cobertura por sucursal, manejar excepciones (sucursal sin stock del producto), y validar direcciones. Es una funcionalidad deseable pero no crítica para la operación inicial.

Implementación futura: Cuando se active el Modo C, la asignación automática evaluará primero cobertura geográfica, luego disponibilidad de stock, y finalmente reglas de negocio (costo de despacho, tiempo estimado). El Modo B se usará como fallback si la asignación automática no encuentra candidata.

Integración con WooCommerce

Cada cambio de estado en el POS actualiza el estado del pedido en WooCommerce mediante la API REST y, opcionalmente, agrega una nota privada visible para el administrador de la tienda.

MVP vs futuro: En el MVP, las rutas de despacho se gestionan manualmente (Modo A o B). La integración con servicios externos de tracking (Chilexpress, Starken, Bluexpress) queda para fases posteriores.
06 · Cumplimiento y Seguridad

Privacidad por Diseño (Ley 21.719)

La Ley 21.719 (modificación a la Ley 19.628 de protección de datos personales en Chile) exige que cualquier sistema que procese datos de ciudadanos chilenos implemente privacidad desde el diseño y por defecto. A continuación, las directrices concretas para el módulo de clientes.

Minimización de datos

El sistema solo debe recolectar los datos estrictamente necesarios para la operación comercial. Los campos obligatorios del formulario de registro son: RUT (para facturación), nombre, email (para comunicación) y teléfono (para despacho). Dirección, fecha de nacimiento, género y otros datos demográficos son opcionales y deben recolectarse con consentimiento explícito adicional.

Cifrado en reposo de campos sensibles

Tres campos se consideran datos sensibles y deben almacenarse cifrados con AES-256 usando una clave por instalación almacenada fuera de la base de datos (variable de entorno APP_DATA_KEY):

  • RUT/DNI: Cifrado con lakm/laravel-encrypted-trait. Se indexa el hash del RUT para búsquedas únicas.
  • Email: Cifrado en reposo. Se almacena un hash SHA-256 en una columna auxiliar para búsquedas por login.
  • Teléfono: Cifrado en reposo. No se indexa para búsqueda.
// Ejemplo: Customer model con cifrado
use LakM\LaravelEncrypted\Traits\HasEncryptedAttributes;

class Customer extends Model
{
    use HasEncryptedAttributes;

    protected \$encrypted = ['rut', 'email', 'phone'];

    // Hash para búsquedas únicas por RUT
    public function setRutAttribute(\$value)
    {
        \$this->attributes['rut'] = encrypt(\$value);
        \$this->attributes['rut_hash'] = hash('sha256', \$value);
    }

    public function scopeFindByRut(\$query, \$rut)
    {
        return \$query->where('rut_hash', hash('sha256', \$rut));
    }
}

Consentimiento explícito

El sistema debe registrar en la base de datos el consentimiento del cliente para cada finalidad de uso de sus datos:

Schema::create('customer_consents', function (Blueprint \$table) {
    \$table->id();
    \$table->foreignId('customer_id')->constrained()->cascadeOnDelete();
    \$table->string('purpose');  // 'marketing', 'data_processing', 'sharing_third_party'
    \$table->boolean('granted');

  • El check de consentimiento no puede venir pre-seleccionado.
  • El cliente puede revocar su consentimiento en cualquier momento desde su perfil.
  • La revocación no invalida el procesamiento previo, pero detiene el uso futuro.

Derecho al Olvido (eliminación / anonimización)

El sistema debe exponer un endpoint (API y UI) para que el cliente solicite la eliminación de sus datos. El proceso sigue estos pasos:

  1. El cliente (o un administrador en su nombre) inicia la solicitud.
  2. El sistema verifica que no existan obligaciones legales que impidan la eliminación (ej. facturas emitidas que requieren retención de datos por 5 años).
  3. Si hay obligaciones, los datos del cliente se anonimizan: se reemplazan nombre, email, teléfono y RUT por valores irreversibles (ej. "Cliente Anonimizado #123", "anon-xxxxx@anonymous.com").
  4. Si no hay obligaciones, se eliminan los registros del cliente y se anonimizan las referencias en ventas históricas.
  5. Se registra en un log de auditoría la acción con fecha, operador y alcance.
// CustomerController
public function requestForget(Customer \$customer)
{
    \$hasInvoices = \$customer->sales()
        ->whereNotNull('dte_uuid')
        ->exists();

    if (\$hasInvoices) {
        // Anonimizar en lugar de eliminar
        \$customer->anonymize();
        return response()->json(['message' => 'Datos anonimizados. Facturas retenidas por obligación legal.']);
    }

    // Eliminación completa
    \$customer->delete();
    return response()->json(['message' => 'Solicitud completada. Datos eliminados.']);
}

Derecho de rectificación

El cliente puede actualizar sus datos personales directamente desde su perfil. Cada cambio queda registrado en una tabla customer_data_changes con el valor anterior, el nuevo valor, la fecha y el IP desde donde se realizó el cambio. Esto permite reconstruir el historial de modificaciones en caso de disputa.

Matriz de Roles MVP

Tres roles acumulativos en el MVP — cada rol hereda los permisos del nivel anterior. El alcance siempre está acotado por sucursal, excepto donde se indique lo contrario.

Principio de alcance acumulativo

Los roles siguen una jerarquía lineal: Cajero → Supervisor → Admin Central. Cada rol incluye todos los permisos del rol anterior más los suyos propios. Esto simplifica la administración: no es necesario gestionar roles compuestos ni permisos individuales.

Implementación técnica

Usamos spatie/laravel-permission con una adaptación multi-sucursal:

  • Los roles se asignan a un usuario en relación con una sucursal mediante la tabla pivot model_branch_roles (user_id, role_id, branch_id).
  • El middleware CheckBranchAccess verifica que el usuario tenga el rol requerido en la sucursal activa de la sesión.
  • Los permisos globales (Admin Central) se asignan sin sucursal (branch_id = null).
  • Para la jerarquía acumulativa, se usa un RoleHierarchyService que permite a un rol ejecutar Gates definidos para roles inferiores.

07 · Preguntas Abiertas

Webpay Físico

La integración con Webpay Plus Presencial (terminal POS físico de Transbank) queda fuera del MVP pero es la funcionalidad más solicitada por los clientes.

¿Qué implica?

Webpay Plus Presencial requiere un terminal POS físico conectado por puerto serie o Bluetooth a un computador con el sistema POS. Transbank provee un SDK (generalmente DLL en Windows o JAR en Java) que debe ser llamado desde el sistema para iniciar la transacción.

Desafíos técnicos

  • Compatibilidad con Laravel (PHP): El SDK de Transbank es para Java/.NET. Habría que crear un bridge (ej. microservicio en Node.js o contenedor Java) que se comunique con Laravel vía socket local o API HTTP.
  • Certificación: Transbank exige certificación para cada integración. El proceso puede tomar 2-4 semanas.
  • Multi-plataforma: El POS corre en navegador (cualquier SO), pero el bridge con el terminal POS es dependiente del SO del operador.
  • Sin demonios en cPanel: El bridge sería un proceso externo que debe ejecutarse en la máquina del operador, no en el servidor.

Arquitectura propuesta (futura)

┌─────────────┐    WebSocket     ┌──────────────┐    Puerto Serie    ┌──────────┐
│  Laravel POS │◄──────────────►│  Bridge POS   │◄────────────────►│ Terminal  │
│  (Navegador) │    localhost    │  (Node.js)    │    / Bluetooth    │ Transbank │
└─────────────┘    :6001        └──────────────┘                   └──────────┘
Decisión pendiente: Evaluar si se usa la nueva API REST de Transbank (Webpay API 2.0) que permite integración sin SDK propietario, mediante QR dinámico y pago desde el teléfono del cliente, eliminando la necesidad del bridge local.
07 · Preguntas Abiertas

Facturación DTE

La facturación electrónica chilena (DTE) es obligatoria para la mayoría de los comercios. Su implementación requiere integración con el SII o un proveedor autorizado.

Contexto legal

Chile exige emisión de Documentos Tributarios Electrónicos (DTE) desde 2018 para la mayoría de los contribuyentes. Los tipos de documento relevantes son:

  • Boleta Electrónica: Para consumidores finales (sin RUT).
  • Factura Electrónica: Para empresas (con RUT).
  • Nota de Crédito/Débito: Para correcciones.

Opciones de integración

ProveedorTipoVentajasDesventajas
SII Directo (libre)API SOAPSin costo por transacciónIntegración compleja, requiere firma electrónica, certificación SII
PayMeAPI RESTSDK moderno, soporte local, precios competitivosCosto mensual + por documento
EfactAPI RESTAmplia adopción en Chile, buen soporteCosto más alto
SimpleFactAPI RESTEconómico, API modernaMenos trayectoria

Preguntas abiertas para decisión

  • ¿Se factura desde el POS o desde WooCommerce? Opción A: El POS emite la boleta/factura al momento de la venta presencial. Opción B: WooCommerce emite los DTE de pedidos web. Lo ideal sería ambos, pero duplica la integración.
  • ¿Timing de emisión? Inmediata (al pagar) vs. diferida (al despachar). La ley permite ambas, pero la inmediata es mejor para el cliente.
  • ¿Múltiples sucursales con un solo RUT? Si el cliente tiene un RUT para todas las sucursales, los DTE se emiten con ese RUT pero deben identificar la sucursal emisora en el detalle.
  • ¿RUT de cliente obligatorio? Para boletas no es obligatorio; para facturas sí. El POS debe manejar ambos flujos.
Decisión pendiente: Se recomienda integrar con un proveedor tipo PayMe para la Fase 2, ya que reduce drásticamente el tiempo de implementación comparado con la integración directa al SII. El costo por transacción se traslada al cliente como costo operacional.
08 · Estructura y Operación Interna

Core Backend: Estructura de Directorios

El proyecto rechaza la estructura MVC plana de Laravel (app/Models, app/Http/Controllers) en favor de una organización por dominios (DDD ligero). Esto aísla la lógica de negocio para un desarrollador que trabaja solo y facilita migraciones futuras.

Decisión arquitectónica: No usar app/Models/ como cajón de sastre. Cada módulo vive en app/Domain/<Nombre>/ con sus propios modelos, acciones, jobs y reglas de negocio. Esto permite que, en un escenario de crecimiento, un dominio como Sync pueda extraerse a un microservicio independiente con mínimo esfuerzo.

Principios de la estructura

  • Controllers delgados: Cada método de controlador valida la request, llama a una Action y retorna una respuesta. No contiene lógica de negocio.
  • Actions con una sola responsabilidad: Cada Action ejecuta una tarea de dominio específica. Son invocables desde controllers, jobs o comandos.
  • Jobs en el dominio que los origina: SyncStockToWooCommerce vive en Domain/Sync/Jobs/, no en app/Jobs/.
  • Modelos auto-contenidos: Cada modelo incluye sus relaciones, scopes y atributos calculados. No hay repositorios externos que filtren consultas.
  • Enums y DTOs tipifican el dominio: Los enums reemplazan strings mágicos. Los DTOs transportan datos entre capas sin acoplar a Eloquent.

Beneficio para migraciones futuras

Si el negocio crece y se decide extraer la sincronización a un microservicio independiente, el directorio app/Domain/Sync/ se copia a un nuevo proyecto Laravel con cambios mínimos. Las Actions y Jobs ya están encapsuladas; solo hay que reemplazar el WooCommerceApiClient por llamadas HTTP al nuevo servicio. El resto del sistema (Inventory, Sales, Identity) ni se entera del cambio.

Consejo para el desarrollador: No te sientas obligado a crear todas las carpetas de una vez. Empieza con Domain/Sales para el POS y Domain/Inventory para el stock. A medida que el código crezca, extraé Domain/Sync y Domain/Reporting. La estructura debe evolucionar con el sistema, no definirse rígidamente el día cero.
08 · Estructura y Operación Interna

Reportería y Rendimiento

Regla de oro: prohibido calcular reportes en tiempo real sobre tablas transaccionales. El rendimiento del POS es la prioridad máxima; la latencia en reportes es una decisión de diseño aceptada.

Riesgo crítico: Una consulta SELECT SUM(quantity) FROM stock_ledger WHERE ... sobre 500k movimientos mientras un cajero intenta cobrar en el POS puede congelar la base de datos compartida de cPanel. La pre-agregación no es optativa: es obligatoria para la viabilidad del sistema en producción.

Estrategia de pre-agregación

En lugar de consultar las tablas transaccionales (stock_ledger, sales) para cada reporte, se ejecutan Jobs programados que consolidan los datos en tablas de resumen. Los reportes leen estas tablas pre-agregadas.

Job: AggregateDailySales

Se ejecuta cada 30-60 minutos. Agrupa las ventas del día por sucursal, método de pago y categoría de producto, y escribe los totales en daily_sales_summaries.

// app/Domain/Reporting/Jobs/AggregateDailySales.php
public function handle(): void
{
    // 1. Obtener ventas del día actual que aún no han sido agregadas
    \$lastAggregation = DailySalesSummary::max('aggregated_until')
        ?? now()->subDay();

    \$sales = Sale::where('created_at', '>', \$lastAggregation)
        ->where('status', '!=', 'cancelled')
        ->get();

    // 2. Agrupar por sucursal × método de pago × categoría
    \$groups = \$sales->groupBy(fn(\$s) => [
        'branch_id'    => \$s->branch_id,
        'payment_method' => \$s->payment_method,
        'date'         => \$s->created_at->toDateString(),
    ]);

    // 3. Upsert masivo en tabla de resumen
    foreach (\$groups as \$key => \$group) {
        DailySalesSummary::upsert([
            'branch_id'      => \$group[0]->branch_id,
            'payment_method'  => \$group[0]->payment_method,
            'date'           => \$group[0]->created_at->toDateString(),
            'total_sales'    => \$group->count(),
            'total_revenue'  => \$group->sum('total'),
            'aggregated_until' => now(),
        ], ['branch_id', 'payment_method', 'date']);
    }
}

Job: SnapshotStock

Se ejecuta cada 15-30 minutos. Toma una foto del stock actual de cada producto por sucursal y la almacena en stock_snapshots. Esto permite consultar el stock histórico sin sumar el ledger completo.

// app/Domain/Reporting/Jobs/SnapshotStock.php
public function handle(): void
{
    \$products = Product::with('branches')->get();

    \$snapshots = [];
    foreach (\$products as \$product) {
        foreach (\$product->branches as \$branch) {
            \$snapshots[] = [
                'product_id'     => \$product->id,
                'branch_id'      => \$branch->id,
                'stock_quantity' => \$product->stockEnSucursal(\$branch),
                'snapshot_at'    => now(),
            ];
        }
    }

    StockSnapshot::insert(\$snapshots);  // insert masivo
}

Latencia como decisión de diseño

El panel de reportes del MVP muestra datos con una latencia de hasta 60 minutos. Esto es intencional. El cajero no puede esperar a que una consulta pesada termine para completar una venta. El administrador que revisa reportes puede tolerar datos de "hace 30 minutos".

Si en Fase 2 se requiere reportes en tiempo real, se implementa una capa de caché (Redis si el hosting lo soporta, o tabla denormalizada branch_product_stock) que se actualiza mediante eventos de dominio, no por consultas directas al ledger.

Métrica de rendimiento objetivo: Toda transacción POS debe completarse en <500ms. Las consultas de reportes no deben degradar este tiempo. Si una consulta de reporte tarda 10 segundos pero no afecta al POS, es aceptable. Si una consulta de reporte hace que una transacción POS tarde 2 segundos, hay que corregirla.
09 · Frontera del MVP y Backlog Fase 2

Límites y Exclusiones del MVP

Esta página define explícitamente qué funcionalidades están fuera del alcance de la primera versión entregable (MVP) y documenta su alcance conceptual para el desarrollo futuro. Ninguna de estas funcionalidades debe implementarse en la fase MVP.

1. Pagos Integrados (Webpay POS / Transbank)

Situación en el MVP: El POS registra los pagos de forma manual e informativa. El cajero selecciona "Efectivo", "Débito" o "Crédito" como método de pago, pero la integración con el terminal físico de Transbank no existe. El flujo de pago es: el cajero cobra en el POS, el sistema registra el método, y el cajero opera el terminal POS físico de forma independiente.

Por qué no está en el MVP:

  • Trámites comerciales: La integración con Transbank requiere un contrato comercial, certificación técnica (2-4 semanas) y, en algunos casos, un POS físico proporcionado por el banco. Esto no es un obstáculo técnico sino comercial y de tiempo.
  • Bridge local: El SDK de Transbank es Java o .NET, no PHP. Se necesita un proceso externo (bridge Node.js o Java) que se comunique con el terminal por puerto serie o Bluetooth. Este bridge debe ejecutarse en la máquina del operador, no en el servidor, y orquestarlo desde Laravel requiere un canal de comunicación (WebSocket local o API HTTP en localhost).
  • Multi-plataforma: El bridge sería específico del SO del operador (Windows .dll vs Linux .so), duplicando el esfuerzo de desarrollo y testing.

Alcance conceptual para Fase 2:

  • Evaluar Webpay API 2.0 (QR dinámico) que permite al cliente pagar desde su teléfono sin terminal POS físico, eliminando la necesidad del bridge.
  • Si se requiere terminal físico, implementar un bridge Node.js con comunicación vía WebSocket (localhost) hacia el navegador del POS.
  • La certificación Transbank corre por cuenta del cliente, no del desarrollador.

2. Facturación Electrónica (DTE / SII)

Situación en el MVP: El sistema emite un comprobante interno (ticket o comprobante no tributario) al finalizar cada venta. Este comprobante cumple como respaldo de la operación pero no tiene validez fiscal.

Por qué no está en el MVP:

  • Obligación legal vs complejidad técnica: Emitir DTE requiere integración con el SII (SOAP XML con firma electrónica) o con un proveedor autorizado (PayMe, LibreDTE, Haulmer). Cada proveedor tiene su propia API, modelo de datos y costos asociados.
  • Decisión de proveedor pendiente: El cliente debe elegir el proveedor de facturación. No es una decisión técnica sino comercial. El desarrollador no debe forzar una elección en el MVP.
  • Múltiples flujos: Boleta electrónica (sin RUT) vs factura electrónica (con RUT). Notas de crédito/débito. Cada flujo duplica la lógica de integración.

Alcance conceptual para Fase 2:

  • Integrar con un proveedor API REST (PayMe recomendado por SDK moderno y soporte local).
  • El POS emite boleta/factura al momento del pago. WooCommerce emite DTE para pedidos web al confirmar despacho.
  • Implementar tabla sales.dte_uuid para trackear el documento fiscal asociado a cada venta.
  • La firma electrónica del cliente se almacena de forma segura en la configuración del sistema.
Importante: La emisión de comprobantes no tributarios en el MVP es legal siempre que el cliente emita los DTE correspondientes por otro medio (ej. WooCommerce con plugin de facturación, o facturación manual). El POS no reemplaza la obligación fiscal del cliente; es responsabilidad de este último asegurar el cumplimiento tributario.

3. Modo Offline (PWA / IndexedDB)

Situación en el MVP: El POS requiere conexión continua al servidor (cPanel). Cada transacción se registra mediante Inertia.js hacia Laravel. Si la conexión se pierde, el POS deja de operar.

Por qué no está en el MVP:

  • Complejidad de sincronización bidireccional: Implementar un caché local (IndexedDB en el navegador) que almacene transacciones offline y las sincronice al reconectar requiere un motor de conflictos, detección de cambios, y lógica de reconciliación que duplica gran parte del backend en el frontend.
  • Caché de catálogo: El navegador debe almacenar localmente el catálogo de productos, precios y stock. Para catálogos grandes (>5000 productos), esto implica manejar cuotas de almacenamiento, versionado de datos y actualización diferencial.
  • Colisiones y consistencia: Dos cajeros offline en la misma sucursal podrían vender el mismo último producto. Resolver esa colisión post-reconexión es un problema de dominio que requiere una estrategia de "último en llegar pierde" o "reserva temporal".

Alcance conceptual para Fase 3:

  • Implementar Service Worker + IndexedDB como capa de persistencia local.
  • Jobs de sincronización en el frontend que empujen transacciones acumuladas al servidor al recuperar la conexión.
  • Estrategia de resolución: "primero en llegar, primero en servirse" con alerta al administrador sobre colisiones resueltas.
  • El catálogo se cachea con versionado: cada producto tiene un version_id que el frontend compara para saber si necesita actualizar su copia local.
Decisión consciente: El MVP sacrifica el modo offline para reducir la complejidad y acelerar la entrega. La mayoría de los comercios chilenos tienen conectividad estable en sus sucursales urbanas. El modo offline es una funcionalidad diferencial para clientes con sucursales en zonas rurales o con conectividad intermitente.

4. Enrutamiento Geográfico (Modo C — Despacho)

Situación en el MVP: Los pedidos web se asignan a sucursal mediante Modo A (sucursal fija del CanalWoo) o Modo B (selección manual por supervisor). No hay asignación automática por geolocalización.

Por qué no está en el MVP: Ver sección 5.1 Rutas de Despacho. Requiere mantener una base de datos de comunas, tabla de cobertura por sucursal, lógica de excepción por falta de stock, y validación de direcciones. Es deseable pero no crítico para la operación inicial.

Alcance conceptual para Fase 2:

  • Tabla branch_coverage con (branch_id, comuna, activo, prioridad).
  • Algoritmo de asignación: cobertura geográfica → disponibilidad de stock → costo de despacho → tiempo estimado.
  • Fallback a Modo B si la asignación automática no encuentra candidata.
  • API externa de geocodificación (Google Maps / OpenStreetMap) para validar direcciones.

5. Integración con servicios de tracking (Chilexpress, Starken, Bluexpress)

Situación en el MVP: No hay tracking externo. El estado de despacho se actualiza manualmente en el POS y se refleja en WooCommerce.

Por qué no está en el MVP: Cada courier tiene su propia API de tracking con distintos formatos y requerimientos de autenticación. La integración es puramente operativa y no afecta la transaccionalidad del negocio.

Alcance conceptual para Fase 2: Tabla shipping_providers con configuración de API por courier. Webhook receptor para actualizar estado automáticamente cuando el courier escanea el paquete.
